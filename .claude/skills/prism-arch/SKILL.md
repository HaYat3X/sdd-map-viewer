# prism-arch

## 責務

「What（要件）を How（設計）に変換する」。

ただし cc-sdd が実装の余地を持てるよう、**コンポーネント境界と方針を決めるレベル**に留める。
実装詳細（APIスキーマ・DBスキーマ・UI構成）は cc-sdd に任せる。

---

## 実行手順

### Step 1: 入力ファイルの確認

以下が存在するか確認：

```
docs/charter.md                        ← ✅ Confirmed 制約・体制
docs/requirements/overview.md          ← スコープ・非機能要件・機能一覧
docs/requirements/req_*.md             ← 機能ごとの要件（全ファイル）
docs/spikes/*/spike_*.md（ある場合）   ← スパイク結果
```

すべて存在しない場合はエラーで停止：

```
「docs/requirements/ が見つかりません。
 まず /prism-req を実行してください。」
```

### Step 2: 制約の確認

charter.md の ✅ Confirmed から以下を抽出する：

```
- 技術・インフラ制約（MCP制限、ローカル動作など）
- 予算・スケジュール制約
- セキュリティ・コンプライアンス要件
- 利用可能なリソース・体制
```

スパイク結果がある場合、docs/spikes/_/spike\__.md の「調査結果」「設計への反映」セクションを読む。

### Step 3: システム境界の定義

`docs/architecture/system-context.md` を生成する。

```
定義する内容:
  - このシステムの外側にあるもの（外部システム・ユーザー・データソース）
  - このシステムが担う範囲
  - 外部との接続方式（API / ファイル / DB共有 etc.）

参考資料:
  - charter.md の Goals / Non-goals
  - requirements/overview.md のスコープ
```

### Step 4: コンポーネント分割

`docs/architecture/container-diagram.md` を生成する。

```
分割の基準:
  - デプロイ単位が異なる → 別コンテナ
  - 責務が明確に異なる → 別コンポーネント
  - 技術スタックが異なる → 別コンテナ

各コンポーネントに記載する情報:
  - 名前・責務
  - 使用技術（フレームワーク・言語）
  - 他コンポーネントとの接続方式
  - 主要な req_*.md との対応
```

### Step 5: 技術スタック選定

`docs/architecture/tech-stack.md` を生成する。

```
選定する内容:
  - フロントエンド（フレームワーク・状態管理・UIライブラリ）
  - バックエンド（言語・フレームワーク・ORM）
  - データストア（DB種別・キャッシュ・ファイルストレージ）
  - インフラ（ホスティング・CI/CD・監視）
  - 外部サービス（認証・通知・地図 etc.）

各選定に記載する情報:
  - 何を選んだか
  - なぜ選んだか（charter.md の制約・要件との整合）
  - 何を諦めたか（トレードオフ）
  - スパイク結果がある場合はその根拠を引用
```

### Step 6: データフロー設計

`docs/architecture/data-flow.md` を生成する。

```
設計する内容:
  - 主要なユーザーアクションとデータの流れ
  - データの永続化タイミング・場所
  - 外部システムとのデータ同期方式
  - 状態遷移が複雑な機能の流れ（フローチャート）
```

### Step 7: リスク整理

`docs/architecture/risks.md` を生成する。

```
記載する内容:
  - 技術リスク一覧（スパイク由来 + 設計時に発見したもの）
  - 各リスクのレベル（🔴高 / 🟡中 / 🟢低）
  - 対応方針（軽減策 / 受容 / 回避）
  - 未解決の⚠️がある場合はその旨を明記
```

### Step 8: 実装順序の決定

`docs/architecture/implementation-order.md` を生成する。これが cc-sdd への投入計画になる。

```
決定の基準:
  1. 依存関係（A が完成しないと B が作れない）
  2. リスクの高いものを先に（リスクが後になると手戻りが大きい）
  3. ユーザーへの価値が早く出るもの優先（デモ・検証に使えるもの）

記載形式:
  投入順序 | req_*.md | 依存する機能 | 理由
```

### Step 9: 設計の品質チェック

生成完了後、以下を自己チェックする：

```
□ charter.md の ✅ Confirmed 制約をすべて反映しているか
□ スパイク結果の「設計への反映」を参照しているか
□ cc-sdd が実装できる粒度まで設計を落とせているか
□ implementation-order.md が requirements/req_*.md の全ファイルをカバーしているか
□ 未解決のリスクが risks.md に記録されているか
```

### Step 10: 完了サマリーを出力して停止

```
## prism-arch 完了

アーキテクチャ設計書を生成しました。

docs/architecture/
  ├── system-context.md
  ├── container-diagram.md
  ├── tech-stack.md
  ├── data-flow.md
  ├── risks.md
  └── implementation-order.md

🔴 高リスク: {N}件
🟡 中リスク: {N}件

---
## 次のステップ

1. 設計書の内容を確認してください

2. 未解決のリスク（⚠️）がある場合は、追加のスパイク実施を検討：
   → /prism-spike [YY-MM-DD]

3. 設計が確定したら cc-sdd へ投入開始：
   → implementation-order.md に従って要件ファイルを順番に投入

4. 設計中に新しい設計判断が出た場合は ADR を記録：
   → /prism-adr {タイトル}
```

完了後は必ず**停止**する。自動で次のコマンドを実行しない。

---

## 注意事項

- **決めないこと（cc-sdd に委ねること）**

```
決めない:
  - REST API のエンドポイント設計
  - データベースのカラム設計
  - UI コンポーネントの構成・スタイリング
  - エラーハンドリングの実装方式
  - テストコードの構成

コメントは可（制約として書かない）:
  - 「認証トークンはヘッダーで渡すことを想定」← 方針は書いていい
  - 「DBはPostgreSQLを想定。スキーマはcc-sddで設計」← このレベルでOK
```

- charter.md の 🔵 Draft / 🟡 Review 項目に依存する設計は避ける
  → 前提が変わるリスクが高い
- スパイク結果の「調査結果」「設計への反映」を必ず参照する
  → スパイク成果を活かさないと意味がない
