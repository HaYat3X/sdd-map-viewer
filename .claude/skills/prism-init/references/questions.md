# 情報収集 質問ガイド

## 基本方針

- 質問は **1〜2問ずつ** 出す。一気に全部聞かない
- 「未定」「不明」「まだ決まってない」は記録してよい。無理に引き出さない
- 業界・業種が特定できたら、以降の質問をその文脈に合わせて言い換えてよい
- 回答が曖昧でも「ありがとうございます、🔵 Draft として記録します」と進む

---

## 質問グループ1: プロジェクトの基本情報

```
Q1-1. プロジェクト名（または仮称）を教えてください。

Q1-2. このプロジェクトで「何を実現したいか」を一言で教えてください。
      （例: 社内の設備管理を紙からシステムに移行したい）

Q1-3. なぜ今このプロジェクトが必要になったのか、背景・きっかけを教えてください。
```

→ Goals / Background を 🔵 Draft で埋めながら次へ。

---

## 質問グループ2: スコープと対象ユーザー

```
Q2-1. このシステム・サービスを使う人は誰ですか？
      （例: 現場の保全担当者、管理者、一般ユーザー）

Q2-2. 今回「やること」と「やらないこと」のイメージはありますか？

Q2-3. 既存のシステムや業務フローと連携しますか？
```

→ User Stories / Non-goals の 🔵 Draft を埋める。

---

## 質問グループ3: 規模・予算・スケジュール

```
Q3-1. 予算の概算はありますか？
      （例: 〜500万、1000万前後、未定）

Q3-2. リリースの目標時期はありますか？

Q3-3. ユーザー数・データ量の規模感はありますか？
```

→ Constraints の 🔵 Draft を埋める。

---

## 質問グループ4: 関係者・体制

```
Q4-1. プロジェクトの意思決定者は誰ですか？

Q4-2. 主な関係者・部署と、それぞれの役割を教えてください。

Q4-3. 開発・実装を担当するチームはどちらですか？
```

→ Stakeholders の 🔵 Draft を埋める。

---

## 質問グループ5: 技術・インフラ制約

```
Q5-1. インフラ・環境に制約はありますか？
      （例: オンプレ必須、クラウド不可）

Q5-2. セキュリティ・情報管理に関する制約はありますか？

Q5-3. 使用デバイス・対応ブラウザの指定はありますか？
```

→ Constraints（技術・インフラ）に追記。

---

## 質問グループ6: 成功条件・仮説・リスク

```
Q6-1. このプロジェクトが成功したと言える状態はどんな状態ですか？
      定量的に表現できると理想的です。

Q6-2. このプロジェクトで「最も確かめたいこと・仮説」はありますか？
      （例: 現場担当者がスマホで使えれば点検効率が上がるはず）

Q6-3. 今時点で気になっていること・リスクはありますか？
      過去に似たプロジェクトで失敗したことがあれば教えてください。
```

→ Success Metrics / Hypotheses / Open Questions の 🔵 Draft を埋める。

---

## 未定だった場合の記録方法

| 項目         | 記録                                      |
| ------------ | ----------------------------------------- |
| 予算         | `🔵 未定（ヒアリング後に確定予定）`       |
| スケジュール | `🔵 未定（〇〇が決まってから確定）`       |
| ユーザー数   | `🔵 不明（調査中）`                       |
| インフラ制約 | `🔵 不明（情報システム部への確認が必要）` |

⚠️ マークは「矛盾あり・要確認」の項目だけにつける。
未定・不明は ⚠️ ではなく 🔵 Draft として記録する。
