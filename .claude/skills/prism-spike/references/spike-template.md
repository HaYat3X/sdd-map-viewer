# スパイクタスクテンプレート

`docs/spikes/{YY-MM-DD}/spike_{N}_{キーワード}.md` を生成する際のフォーマット。

---

```md
# スパイク {N}: {調査対象を端的に表すタイトル}

## リスクレベル

🔴 必須 / 🟡 推奨 ← どちらか削除

## 発生源

{analysis.md のどの要求・潜在要求から発生したか}

例:

- 要求: 「設備を地図で表示したい」
- 潜在要求（仮説）: 「設備データが10万件以上ある可能性」

## なぜスパイクが必要か

{この不確実性を放置すると何が起きるか}

例:
設備件数が10万件を超える場合、クライアントサイドで全件レンダリングすると
初期表示に数十秒かかる可能性がある。
タイル分割・クラスタリング・サーバーサイドフィルタリングなどの
アーキテクチャ選択が変わるため、要件定義完了前に方針を決める必要がある。

## 技術リスクの仮説

{何がわからないと何が破綻するか、具体的に}

例:

- 設備データの実件数が不明
- 地図ライブラリの最大表示件数が不明
- バックエンドのフィルタリングAPIが必要かどうか不明

## 調査方法

{どうやって調べるか}

例:

1. 顧客に設備データの件数・増加ペースを確認（ヒアリング or データ提供依頼）
2. 想定件数で地図ライブラリ（MapLibre / Leaflet）の PoC を実施
3. 10万件でのレンダリング時間・メモリ使用量を計測

## 判断基準

{この結果が出たら「実現可能」「方針A」「方針B」とする}

例:

- 件数 < 1万件 → クライアントサイド全件レンダリングで対応可能
- 件数 1万〜10万件 → クラスタリング表示を採用
- 件数 > 10万件 → サーバーサイドタイリング必須

## タイムボックス

{推奨調査時間}

例: 0.5日（PoC実施含む）

## 担当

🔵 未定 ← スパイク実施後に更新

## ステータス

🔵 Draft

## 調査結果

（スパイク完了後に記録）

## 設計への反映

（prism-arch が参照して記録）
```
