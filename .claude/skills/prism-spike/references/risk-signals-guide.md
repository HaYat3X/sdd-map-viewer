# リスクシグナル判断ガイド

hearing-analyst の出力から技術リスクを検出するための判断基準。

---

## 🔴 スパイク必須シグナル

以下のいずれかが要求・潜在要求に含まれている場合、スパイク必須と判断する。

### データ規模の不確実性

```
シグナルワード: 「大量」「多数」「全件」「一覧」「履歴」「ログ」
判断基準: 件数が不明かつ数万件を超える可能性がある
スパイクの目的: データ量の見積もり・表示戦略の feasibility 確認

例:
  「設備を地図で表示したい」 + 潜在要求「10万件以上ある可能性」
  → 件数確認・地図パフォーマンス PoC が必要
```

### リアルタイム性の要求

```
シグナルワード: 「リアルタイム」「即時」「自動更新」「ライブ」「最新」
判断基準: ポーリング/WebSocket/SSE のいずれかが必要になる可能性
スパイクの目的: 更新頻度・許容遅延の確認、アーキテクチャ方針の決定

例:
  「センサーの状態を常に確認したい」
  → 更新頻度・許容遅延・インフラ対応可否の確認が必要
```

### 外部システム連携の詳細不明

```
シグナルワード: 「連携」「インポート」「エクスポート」「既存システム」「基幹」
判断基準: 連携先のAPI/データ形式が未確認
スパイクの目的: API仕様・認証方式・データ形式の確認

例:
  「既存の台帳システムからデータを取り込みたい」
  → 台帳システムのAPI有無・データ形式確認が必要
```

### 認証・認可の方式が未定

```
シグナルワード: 「ログイン」「権限」「管理者」「役割」「アクセス制限」
判断基準: 認証基盤（既存SSO/新規構築/外部サービス）が未定
スパイクの目的: 認証方式の選定・既存インフラとの整合確認

例:
  「社員だけがアクセスできるようにしたい」
  → 社内SSO連携の可否確認が必要
```

### オフライン対応の可能性

```
シグナルワード: 「現場」「電波が悪い」「ネットワーク不安定」「持ち歩き」
判断基準: オフライン動作・データ同期が必要になる可能性
スパイクの目的: オフライン要件の有無確認・技術選定への影響評価

例:
  「現場でタブレットから入力したい」
  → オフライン要件の明確化が必要
```

## 🟡 スパイク推奨シグナル

要件定義は進められるが、実装前に確認しておくべき項目。

### パフォーマンス要件の数値が未定

```
シグナルワード: 「速く」「重くない」「サクサク」「レスポンスが良い」
判断基準: SLA/応答時間の数値が顧客から出ていない
スパイクの目的: 許容応答時間の確認・ボトルネック候補の洗い出し
```

### ファイル・メディアの扱い

```
シグナルワード: 「写真」「画像」「ファイル」「PDF」「アップロード」
判断基準: ファイルサイズ・形式・保存先が未定
スパイクの目的: ストレージ方式の選定・容量見積もり
```

### 通知・メール送信

```
シグナルワード: 「通知」「メール」「アラート」「お知らせ」
判断基準: 通知先・タイミング・インフラの可否が未確認
スパイクの目的: 通知基盤の選定・既存インフラとの整合確認
```

---

## 🟢 スパイク不要（記録のみ）

以下に該当する場合はスパイク不要。analysis の内容を参照して判断する。

```
- 一般的なCRUD（作成・読み取り・更新・削除）の範囲内
- 既知の技術スタックで対応実績がある
- データ件数が明確かつ数千件以内
- 既存の認証基盤を使うことが確定している
- 外部連携なし・スタンドアロン動作
```

## 判断に迷ったとき

「この不確実性が解消されないと、要件定義の前提が変わる可能性があるか？」

→ YES → 🔴
→ 変わるかもしれないが要件定義は進められる → 🟡
→ NO → 🟢
