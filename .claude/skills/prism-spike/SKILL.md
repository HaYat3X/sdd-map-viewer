# prism-spike

## 責務

「この要求は本当に作れるか？」を調べるための調査タスクを生成する。

**決めない。調べるだけ。**

スパイクで出た結論は arch-designer が設計に使う。
このコマンドは「何を調べるか」を定義するだけで、設計判断は行わない。

---

## 実行手順

### Step 1: 入力確認

ユーザーから以下を受け取る：

```
/prism-spike [YY-MM-DD]

[YY-MM-DD] = ヒアリング分析日（例: 26-04-15）
```

入力ファイル：`docs/analysis/{YY-MM-DD}/analysis.md`
出力先：`docs/spikes/{YY-MM-DD}/spike_{N}_{キーワード}.md`

### Step 2: 前提ファイルの確認

```
docs/charter.md が存在するか確認
  ✅ 存在 → ✅ Confirmed 項目を参照

docs/analysis/{YY-MM-DD}/analysis.md が存在するか確認
  ✅ 存在 → Step 3 へ
  ❌ 存在しない → エラーで停止
```

analysis.md が存在しない場合：

```
「docs/analysis/{YY-MM-DD}/analysis.md が見つかりません。
 まず /prism-analyze [YY-MM-DD] を実行してください。」
```

### Step 3: 技術リスクシグナルのスキャン

analysis.md の以下のセクションを読み込む：

```
- 要求分解テーブル（要求・潜在要求の列）
- 矛盾・認識ズレ（⚠️マーク箇所）
- 技術的不確実性（スパイク候補）セクション
```

各要求・潜在要求に対して、`references/risk-signals-guide.md` の判断基準を適用する。

### Step 4: リスクレベルの分類

スキャン結果を3段階に分類する：

```
🔴 スパイク必須
  → 要件定義を完了する前に調査が必要
  → 調査結果次第でスコープ・方針が変わる可能性がある

🟡 スパイク推奨
  → 要件定義は進められるが、並行調査を推奨
  → 実装前に確認しておくと安全

🟢 スパイク不要
  → 既知技術で対応可能
  → 記録だけして終了
```

### Step 5: スパイクタスク生成

🔴・🟡 に該当する項目について、`references/spike-template.md` に従って
`docs/spikes/{YY-MM-DD}/spike_{N}_{キーワード}.md` を生成する。

連番は 1 から開始。同一日付で複数スパイクがある場合は spike_1, spike_2, ... と採番する。

### Step 6: 出力フォルダの準備

```
docs/spikes/{YY-MM-DD}/ フォルダを作成（存在しない場合）
```

### Step 7: 完了サマリーを出力して停止

スパイクが存在する場合：

```
## prism-spike 完了

docs/spikes/{YY-MM-DD}/ にスパイクタスクを生成しました。

🔴 スパイク必須: {N}件
  - spike_1_{キーワード}.md
  - spike_2_{キーワード}.md

🟡 スパイク推奨: {N}件
  - spike_3_{キーワード}.md

🟢 スパイク不要: {N}件

---
## 次のステップ

1. 🔴 必須スパイクを実施してください
   各spike_*.md に「調査方法」が記載されています

2. スパイク結果を spike_*.md の「調査結果」セクションに記録してください

3. スパイク実施後：
   → /prism-req（要件定義に進む）
   または
   → /prism-hearing [YY-MM-DD]（ヒアリング追加実施）
```

スパイクが不要な場合：

```
## prism-spike 完了

技術リスク・不確実性は検出されませんでした。
既知技術で要件定義・実装を進めることができます。

---
## 次のステップ

→ /prism-req（要件定義に進む）
```

完了後は必ず**停止**する。自動で次のコマンドを実行しない。

---

## 注意事項

- 潜在要求は「仮説」であることを忘れない
  → 潜在要求に対するスパイクは「仮説の検証」として位置づける
  → スパイクで仮説が否定されることもある。それでいい

- スパイクを増やしすぎない
  → 全部調べようとすると前に進めなくなる
  → 🔴 に絞ってスパイクを生成し、🟡 は「推奨」として人間の判断に委ねる

- スパイクは「どう作るか」を決めるタスクではない
  → 「作れるかどうか」「どんな制約があるか」を調べるタスク
  → 設計判断は arch-designer に委ねる
