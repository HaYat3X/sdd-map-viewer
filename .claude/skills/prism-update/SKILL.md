# ステータス管理ルール

charter.md の各項目のステータスを管理するときの判断基準。

---

## ステータスの定義

| ステータス   | 意味                                                         | 更新者           |
| ------------ | ------------------------------------------------------------ | ---------------- |
| 🔵 Draft     | PMが提案ベースで記録した情報。未確認・暫定。                 | PM agent（自動） |
| 🟡 Review    | 人間への確認待ち。「これ確定していいですか？」と問いかけ中。 | PM agent（自動） |
| ✅ Confirmed | 人間が承認した確定情報。後続スキルが前提として使う。         | 人間（承認）     |

---

## Draft に入れてよいもの

- ヒアリング発言から読み取れる情報
- 前回分析（analysis\_\*.md）からの更新候補
- 人間の会話から得られた暫定情報
- 「まだ決まってない」「たぶん〜」という発言

Draft は「仮説」。後続スキルへの影響を考えて書きすぎない。

---

## Review に上げるタイミング

以下の条件が揃ったら 🔵 Draft → 🟡 Review に上げて人間に確認する：

- ヒアリングで明示的な発言・合意があった
- 複数回のヒアリングで同じ内容が繰り返された
- analysis\_\*.md でエビデンスとして記録された情報

**確認の仕方：**

```
「以下の項目を Confirmed にしてよいですか？

  項目: {項目名}
  現在: {Draft の内容}
  更新候補: {新しい内容}
  根拠: {ヒアリング発言 or 明示的合意}

  ✅ Confirmed にする
  🔵 Draft のままにする
  ✏️ 内容を修正する」
```

---

## Confirmed の扱い

- 人間が承認したら即 ✅ Confirmed に更新
- 更新ログに記録する
- 人間の明示的な指示なしに変更しない
- Open Questions から該当項目を削除する

---

## 後続スキルへの参照ルール

```
hearing-preparer     → ✅ Confirmed + 🔵 Draft を参照
                        （Draftは「未確定として扱う」という文脈で渡す）

hearing-analyst      → ✅ Confirmed のみ前提として分析
                        （Draftは「要確認事項」として扱う）

requirements-decomposer → ✅ Confirmed のみ前提として要件化
                           （Draftの項目に依存する要件は書かない）
```

---

## ⚠️ と Draft の違い

```
🔵 Draft  → 情報はある。ただし未確認・暫定。
⚠️        → 矛盾・認識ズレ・情報が不足している。要確認。
```

**例：**

- 「予算: 未定」→ 🔵 Draft（情報なし、でも問題なし）
- 「田中部長は1000万と言ったが、山田部長は500万と言った」→ ⚠️（矛盾あり）
