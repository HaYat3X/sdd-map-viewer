# Requirements Decomposer

ヒアリング分析結果を、cc-sddが実装できる粒度の機能要件MDに変換するスキル。

---

## このスキルの位置づけ

```
{project_root}/docs/analysis/analysis_*.md
    ↓
requirements-decomposer（このスキル）
    ↓
{project_root}/docs/requirements/
├── overview.md             ← スコープ・非機能要件・機能一覧
└── req_[機能名].md × N本   ← cc-sddに1本ずつ食わせる
```

---

## 仕様の粒度について

このスキルが書く「仕様」は**受け入れ条件レベル**に留める。

```
良い例（受け入れ条件）:
- 鉄塔番号を入力すると該当設備にズームする
- 番号が存在しない場合はエラーメッセージを表示する
- 部分一致検索に対応する

書かない（cc-sddに任せる）:
- 検索ボックスはヘッダーに常時表示
- 3文字以上入力でサジェスト表示
- 検索結果は最大20件まで表示
```

理由：UIの詳細・技術的なHowはcc-sddの設計フェーズで決める。
こっちで決めすぎるとcc-sddが設計の余地を失う。

---

## ステップ1: 入力の受け取り

ユーザーに確認する：

```
以下を教えてください：
1. 分析対象のファイルパス
   （例: projects/{project_name}/docs/analysis/analysis_02_20260403.md）
   ※ 複数回のヒアリング分析がある場合は全ファイルを渡してください
2. プロジェクトのルートパス
   （例: projects/{project_name}/）
```

出力先は自動的に `{project_root}/docs/requirements/` になる。
charter.md が存在する場合は読み込んで文脈として参照する。

---

## ステップ2: 機能の洗い出し

`references/decompose-guide.md` を読み込み、分解の基準を確認してから処理する。

### 機能への変換ルール

analysis\_\*.md の要求分解テーブルを読み込み、以下の手順で機能に変換する。

**① 要求を機能にグルーピング**
複数の要求が同一機能で解決できる場合はまとめる。
潜在要求も機能として実現すべきものはここで取り込む。

```
例:
「鉄塔番号で検索したい」（要求）
「設備番号と地理的位置の対応を把握したい」（潜在要求）
    ↓ まとめる
→ 機能: 設備検索
```

**② 機能の命名**
ユーザーの動詞フレーズで命名する。「〜する」「〜を見る」「〜を登録する」。
技術用語は使わない。

**③ overviewとreqの仕分け**

- 機能要件 → 各 `req_*.md` へ
- 非機能要件（性能・セキュリティ・可用性等）→ `overview.md` へ
- スコープ外の将来要件 → `overview.md` の「スコープ外・将来対応」セクションへ

---

## ステップ3: ファイルの生成

`references/output-template-overview.md` と `references/output-template-req.md` を読み込み、テンプレートに従って出力する。

### 生成するファイル・出力先

```
{project_root}/docs/requirements/
├── overview.md
├── req_[機能名].md
└── req_[機能名].md ...
```

機能名はスペースなし・日本語可。例: `req_設備検索.md`、`req_地図表示.md`

---

## ステップ4: ファイル出力と確認

1. `{project_root}/docs/requirements/` に全ファイルを書き出す
   （フォルダが存在しない場合は作成する）
2. ユーザーに以下を伝える：
   - 生成したファイルの一覧
   - 機能数
   - スコープ外に分類したもの（あれば）
   - 受け入れ条件が書けなかった・曖昧なまま残った箇所（あれば）

---

## 注意事項

- 受け入れ条件は「テスターが判定できる状態」まで落とす。「〜できること」で終わる形が目安
- analysis\_\*.md に ⚠️ マークがついている要求は、受け入れ条件を書く前に「この要求はスコープに含めるか？」をユーザーに確認する
- 仕様の詳細（UIレイアウト・技術選定・API設計）は書かない。cc-sddに任せる
- 潜在要求はそのまま要件にしない。「顧客が合意した要求」ベースで書く。ただし潜在要求から導かれる受け入れ条件は追加してよい
