# 要件分解ガイド

requirements-decomposer が要求を機能・要件に変換するときの判断基準。

---

## 機能への変換パターン

### パターン1: 複数要求 → 1機能

複数の要求が同じ機能で解決できる場合はまとめる。

```
要求A: 「鉄塔番号で検索したい」
要求B: 「設備の属性情報を見たい」
潜在要求: 「設備番号と地理的位置の対応を把握したい」
    ↓
機能: 設備検索・詳細表示
受け入れ条件:
- 鉄塔番号で検索すると該当設備にズームできること
- 部分一致検索に対応していること
- 検索結果から設備の属性情報（建設年・材質等）を参照できること
- 番号が存在しない場合はエラーメッセージが表示されること
```

### パターン2: 1要求 → 複数機能

1つの要求が複数の機能に分解できる場合は分ける。

```
要求: 「スマホから現場で使いたい」
    ↓
機能A: レスポンシブ対応（地図表示の一部）
機能B: オフラインキャッシュ
機能C: スマホからの写真登録
```

### パターン3: 潜在要求の扱い

潜在要求はそのまま機能にしない。
ただし「顧客が合意した要求」の受け入れ条件を書くときに潜在要求を参照してよい。

```
潜在要求: 「経年変化・劣化の傾向を写真で追跡・比較したい」
    ↓ そのまま機能にしない
    ↓ でも受け入れ条件に反映する
受け入れ条件（写真登録機能）:
- 設備ごとに複数枚の写真を登録できること
- 写真に撮影日時が自動付与されること       ← 潜在要求から導いた条件
- 過去の写真を日時順に一覧できること       ← 潜在要求から導いた条件
```

## 受け入れ条件の書き方

### 良い受け入れ条件

- テスターが「できた/できなかった」を判定できる
- 主語が明確（誰が・何を）
- 「〜できること」「〜が表示されること」で終わる

```
良い例:
- 鉄塔番号を入力すると該当設備にズームできること
- 電圧階級ごとに異なる色で送電線が表示されること
- VPN接続状態のスマホからアクセスできること

悪い例:
- 検索機能があること（何の検索？）
- 使いやすいこと（判定不能）
- 高速に動作すること（基準不明 → 非機能要件に移動）
```

### 非機能要件に移動するもの

以下は受け入れ条件ではなく `requirements_overview.md` の非機能要件に書く。

- 性能・速度（「〇秒以内に表示」）
- セキュリティ（「VPN経由のみアクセス可」）
- 可用性（「99%稼働」）
- データ量・スケール（「ユーザー50名まで」）

## ⚠️ マーク付き要求の扱い

analysis\_\*.md で ⚠️ がついている要求は、要件に変換する前にユーザーに確認する。

```
確認の仕方:
「[要求名] はanalysis時点で認識ズレ・未確定がありましたが、
 第2回以降のヒアリングで解決しましたか？
 解決していれば要件に含めます。未解決なら今回のスコープ外にします。」
```

---

## スコープ外・将来対応の判断

以下は `requirements_overview.md` の「スコープ外・将来対応」セクションに記載する。

- ヒアリングで明示的に「今回はやらない」と決まったもの
- 予算・スケジュール的に今回は無理と判断されるもの
- 潜在要求のうち顧客が合意していないもの
- 「将来的には〜」という発言が根拠のもの
