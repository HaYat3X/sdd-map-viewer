# 要求分解ガイド

hearing-analyst が要求を分解するときの判断基準。

---

## 各列の定義

| 列           | 定義                                 | 記載基準                                                 |
| ------------ | ------------------------------------ | -------------------------------------------------------- |
| 要求者       | 発言した人の名前・役職               | 不明な場合は「不明」。複数いる場合はカンマ区切り         |
| 要求         | 発言をそのまま要約したもの           | 抽象的でもよい。解釈を加えない                           |
| 要求理由     | なぜその要求が出たか                 | 明示があればそのまま記載。推定の場合は「（推定）」を付記 |
| 潜在要求     | 発言の裏にある本当の目的・期待       | 「（仮説）」を必ず付記。1要求に複数可                    |
| 潜在要求理由 | なぜその潜在要求が存在しそうか       | 根拠がない場合は「根拠なし」と明記                       |
| エビデンス   | 要求・潜在要求の根拠となる発言・資料 | 議事録の該当箇所を引用。口頭の場合はその旨を記載         |
| 仕様         | **空欄のまま**                       | requirements-decomposerフェーズで記載する                |

---

## 要求 vs 潜在要求の判断基準

**要求として記載するもの**

- 顧客が「〜したい」「〜が欲しい」と明言したもの
- 「〜ができない」という現状の不満として表明されたもの

**潜在要求として記載するもの**

- 発言の目的・動機として読み取れるが、明言されていないもの
- 「なぜそれが必要か？」を問い続けたときに出てくる答え
- 複数の要求に共通して見える根本的なニーズ

**判断に迷ったとき**
→ 「顧客はこの要求を通じて、最終的に何を達成したいのか？」を問う。
その答えが発言に含まれていなければ潜在要求。

---

## 潜在要求の深掘り例

```
要求: 「鉄塔の番号で検索したい」（保全担当）

要求理由: 現場で特定の設備をすぐ探したいから

潜在要求①: （仮説）現場作業の開始前に対象設備の位置・状態を素早く把握したい
潜在要求理由: 現場は時間的プレッシャーがある。紙図面を広げて探す時間を省きたいはず

潜在要求②: （仮説）設備番号と地理的位置の対応関係を直感的に把握したい
潜在要求理由: 番号体系と地理的位置が一致していないと、番号検索だけでは現場での判断に使えない
```

## エビデンスの記載方法

```
# 良い例
「鈴木さん発言：衛星写真の方がいい（第1回議事録 2025-03-08）」
「田中部長発言：なるべく早く（第1回議事録 2025-03-08）」

# 根拠が薄い場合
「保全担当の発言より推定（第1回議事録 2025-03-08）」

# 口頭のみの場合
「口頭発言のみ・議事録記録なし（2025-03-08 打ち合わせ）」
```

---

## ⚠️ 矛盾・認識ズレの検出パターン

以下のパターンが見つかったら行に ⚠️ をつけ、確認事項リストに追加する。

- 同一事項について関係者間で異なる発言がある
- 「誰がやるか」が明確でない作業がある
- スコープに含まれるか不明な要求がある（例：ドローンデータ連携）
- 技術的に実現困難と思われる要求がある（例：完全オフライン対応）
- 予算・スケジュールと要求の規模が明らかに合っていない
