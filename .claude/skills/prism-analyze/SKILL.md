# prism-analyze

## 責務

ヒアリング議事録から「要求の裏にある本当の目的」を掘り起こし、
要求分解テーブル・次回確認事項リストを生成する。

スコープが確定するまで仕様は書かない。要求と潜在要求を分離することで、
「言われたものを作ったけど使われなかった」を防ぐ。

---

## 実行手順

### Step 1: 入力確認

ユーザーから以下を受け取る：

```
/prism-analyze [YY-MM-DD]

[YY-MM-DD] = ヒアリング実施日（例: 26-04-15）
```

入力ファイル：`docs/hearings/{YY-MM-DD}/hearing.md`
出力先：`docs/analysis/{YY-MM-DD}/analysis.md`

### Step 2: 前提ファイルの確認

```
docs/charter.md が存在するか確認
  ✅ 存在 → charter を文脈として読み込み

docs/hearings/{YY-MM-DD}/hearing.md が存在するか確認
  ✅ 存在 → Step 3 へ
  ❌ 存在しない → エラーで停止
```

hearing.md が存在しない場合：

```
「docs/hearings/{YY-MM-DD}/hearing.md が見つかりません。
 ヒアリングを実施して hearing.md を埋めてください。」
```

### Step 3: 出力フォルダの準備

```
docs/analysis/{YY-MM-DD}/ フォルダを作成（存在しない場合）
```

### Step 4: 要求の抽出と分解

`references/analysis-guide.md` を読み込み、分解の基準を確認してから処理する。

#### 処理の流れ

**① charter.md から文脈を抽出**

```
- Goals・Non-goals
- Constraints
- Stakeholders
- User Stories
```

charter.md の ✅ Confirmed のみを前提として分析する。
🔵 Draft / 🟡 Review は「要確認」として扱う。

**② hearing.md の「決定事項」「回答」を読み込む**

```
docs/hearings/{YY-MM-DD}/hearing.md から：
  - 決定事項
  - 🔴 必須確認の回答
  - 🟡 重要確認の回答
  - アジェンダ外で出た話題
  - メモ・気になったこと
```

**③ 要求の列挙**

hearing.md の回答セクションから発言を1つずつ取り出す。
1発言 = 1行が基本。同一の目的に向かう複数発言はまとめてよい。

**④ 要求理由の推定**

「なぜその要求が出たか」を発言・文脈から読み取る。
明示されていない場合は「（文脈から推定）」と付記する。

**⑤ 潜在要求の仮説出し**

発言の裏にある「本当に解決したいこと」を推定する。
1つの要求に対して潜在要求が複数あってよい（行を分ける）。
必ず「（仮説）」と明記し、確定事項のように書かない。

**⑥ エビデンスの紐づけ**

要求・潜在要求の根拠となる発言・資料を記録する。
hearing.md 上の該当箇所を引用する形でよい。

**⑦ 矛盾・認識ズレの検出**

```
以下をチェック：
  - hearing.md の「未解決・持ち越し」セクション
  - charter.md との矛盾
  - 前回分析（存在する場合）との矛盾

矛盾がある行には ⚠️ マークをつける
```

### Step 5: 出力ファイルの生成

`references/output-template.md` を読み込み、テンプレートに従って生成：

```
docs/analysis/{YY-MM-DD}/analysis.md
```

日付 [YY-MM-DD] を YYYYMMDD 形式に変換して使用。

### Step 6: 次回ヒアリング確認事項の生成

analysis.md 内の「次回ヒアリング 確認事項リスト」で、以下を分類：

```
🔴 必須確認
  → スコープ・方針に直結する未決定事項
  → スパイク候補の技術リスク

🟡 重要確認
  → 要求の詳細化に必要な項目
  → 潜在要求の仮説検証

🟢 できれば確認
  → 精度向上のための項目
```

### Step 7: 完了サマリーを出力して停止

```
## prism-analyze 完了

docs/analysis/{YY-MM-DD}/analysis.md を生成しました。

要求分解テーブル: {N}行
潜在要求の仮説: {N}件
⚠️ 矛盾・認識ズレ: {N}件

---
## 次回ヒアリング 確認事項

🔴 必須確認: {N}件
🟡 重要確認: {N}件
🟢 できれば: {N}件

---
## 次のステップ

1. analysis.md の内容を確認してください
2. 必要に応じて charter.md を更新してください
   → /prism-update

3. 技術リスク・スパイク候補がある場合：
   → /prism-spike [YY-MM-DD]

4. スコープが固まったら要件定義へ：
   → /prism-req
```

完了後は必ず**停止**する。自動で次のコマンドを実行しない。

---

## 注意事項

- 潜在要求はあくまで仮説。確定要件として扱わない
- エビデンスがない潜在要求は「根拠なし」と明記する
- 仕様列は必ず空欄のまま出力する（prism-req のフェーズで埋める）
- 顧客が言っていないことを要求として追加しない
- 前回の analysis.md が存在する場合は「前回からの更新点」を記録する
