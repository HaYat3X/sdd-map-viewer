# 質問生成ガイド

prism-hearing が charter.md から確認事項を導き出すときの判断基準。

---

## 基本方針

- charter.md の 🔵 Draft・⚠️ Open Questions を確認事項に変換することが最優先
- 前回の analysis\_\*.md が存在する場合は「次回ヒアリング 確認事項リスト」を引き継ぐ
- ドメイン特有の質問は「業界あるある」ではなく、charter.md の文脈から導く
- 1回のヒアリングで消化できる量に絞る。🔴3件・🟡5件・🟢3件が上限の目安
- 「聞けたらいいな」より「聞かないと詰まる」を優先する

---

## charter.md → 確認事項への変換パターン

### パターン1: Draft 項目の確認事項化

charter.md で 🔵 Draft のままの項目は、必ず確認事項に変換する。

```
【予算が Draft の場合】
確認事項（🔴）:
「予算の概算金額を確認する。
 フェーズ分けの可否・スコープの取捨選択の前提になるため今回必ず確定させる」

想定回答パターン:
  A: 具体的な金額が出る → charter.md に反映
  B: 「まだ決まってない」→ いつ決まるか・誰が決めるかを確認
  C: 「言えない」→ 規模感（〜百万 / 〜千万）だけでも確認

【リリース目標が Draft の場合】
確認事項（🔴）:
「リリース目標日を確認する。
 「年度内」などの曖昧な表現は具体的な日付に変換して合意する」

【ユーザー数が Draft の場合】
確認事項（🟡）:
「ユーザー数・利用形態の内訳を確認する。
 ロール別（閲覧のみ・更新権限あり・管理者）の人数まで確認する」

【インフラ制約が不明の場合】
確認事項（🔴）:
「クラウド利用可否を情報システム部に確認してもらう。
 NGの場合はオンプレ構成になり、インフラ設計の前提が変わる」
```

### パターン2: Open Questions ⚠️ を優先度 🔴 に上げる

Open Questions に記載されている項目は必ず 🔴 必須確認に含める。

```
charter.md:
| #   | 内容           | 確認先    | 期限       |
| --- | -------------- | --------- | ---------- |
| 1   | 既存システムとの連携の詳細 | CTO | 本ヒアリング |

  ↓ これはそのまま

今回アジェンダ（🔴 必須確認）:
「既存システムとの連携仕様を確認する。
 APIの有無・データ形式・認証方式が未確定なため確定させる」
```

### パターン3: ステータスの読み方

```
🔵 Draft      → 未確認・暫定。必ず確認事項として聞く
✅ Confirmed  → 既に合意済み。確認不要。念のため確認リストに入れない
🟡 Review     → 前回確認中。今回で Confirmed にすべき（🔴 に上げる）
```

## 前回分析からの引き継ぎ

analysis\_\*.md が存在する場合のロジック：

```
docs/analysis/analysis_{N}_{YYYYMMDD}.md から
「次回ヒアリング 確認事項リスト」を読む
  ↓
【🔴 必須確認】 → 今回のアジェンダ 🔴 に最優先で組み込む
【🟡 重要確認】 → 今回のアジェンダ 🟡 に含める
【🟢 できれば確認】 → 時間があれば 🟢 に追加
```

## ドメイン別の追加質問パターン

### 電力・インフラ系

charter.md に「電力」「設備」「保全」などのキーワードが含まれる場合：

```
- 設備データの機密分類は？（公開不可の情報が含まれるか）
- 法定点検・記録保管の要件はあるか？（記録の改ざん防止・保管期間）
- 停電作業・緊急対応との連携は？（リアルタイム性・可用性要件に影響）
- 協力会社・外部業者がアクセスするか？（権限管理の複雑度に影響）
```

### 社内業務DX系

charter.md に「社内」「業務改善」「DX」などのキーワードが含まれる場合：

```
- 既存の業務フロー（紙・Excel）はどこまで残すか？
- 承認フロー・決裁ルートはあるか？
- 人事異動・組織変更への対応は必要か？
- モバイル利用の有無（出張・在宅での利用）
```

### 顧客向けサービス系

charter.md に「サービス」「ユーザー」「公開」などのキーワードが含まれる場合：

```
- 会員登録・ログインは必要か？
- 個人情報の取り扱いはあるか？（プライバシーポリシー・同意取得）
- 外部サービス連携（決済・SNS・通知）はあるか？
- アクセス集中の想定はあるか？（性能要件に影響）
```

---

## 優先度分類の基準

| 優先度      | 基準                                                                       |
| ----------- | -------------------------------------------------------------------------- |
| 🔴 必須     | スコープ・方針・予算・スケジュールに直結。これが決まらないと設計に進めない |
| 🟡 重要     | 要求の詳細化に必要。次フェーズで詰まるリスクがある                         |
| 🟢 できれば | 精度向上のため。今回聞けなくても次回でも可                                 |

---

## アジェンダの時間配分ガイド

```
60分のヒアリングの場合:
  🔴 必須確認: 25分（1件あたり約8分）
  🟡 重要確認: 20分（1件あたり約4分）
  🟢 できれば: 10分
  ネクストアクション確認: 5分

90分の場合:
  🔴 必須確認: 35分
  🟡 重要確認: 30分
  🟢 できれば: 15分
  ネクストアクション確認: 10分
```

質問を詰め込みすぎず、重要な項目を深掘りする時間を確保すること。

---

## 聞かない・含めない判断

以下に該当する場合は確認事項に含めない：

```
❌ charter.md に記載がない項目
❌ ✅ Confirmed になっている項目
❌ 聞いてもわかりそうにない抽象的な質問
❌ 前回のヒアリングで既に合意・決定している項目
```
